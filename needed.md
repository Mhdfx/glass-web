## 4. DIRECTION DESIGN

Le matériau (miroir, vitrail, dorure, verre) impose une identité visuelle **premium, chaleureuse et raffinée** — à l'opposé du look "utilitaire" des concurrents vitrerie classiques.

**Palette suggérée :**
- Fond principal : noir/anthracite profond ou blanc cassé chaud
- Accent : doré/laiton (`#C9A66B` ou proche) — fait écho aux cadres en laiton et miroirs dorés du catalogue
- Neutre : gris chaud, blanc cassé
- Éviter le bleu/vert "tech" — trop générique pour ce métier

**Typographie :**
- Titres : serif élégante (évoque l'artisanat, le luxe) — ex. Playfair Display, Fraunces, ou similaire
- Texte courant : sans-serif lisible et moderne — ex. Inter, Manrope

**Ambiance visuelle :**
- Beaucoup d'espace négatif, photos en grand format
- Effets de reflet/transparence subtils sur les éléments UI (clin d'œil au matériau verre/miroir)
- Photos toujours au centre — le produit doit parler avant le texte

**Inspiration structurelle :** luxuryhabitat.ma (structure one-page, CTA WhatsApp omniprésent, formulaire de devis) — **à dépasser, pas à copier** (voir section "Niveau d'ambition visé" en haut du document)
**Inspiration visuelle premium à explorer :** sites de miroiteries françaises haut de gamme type miroiterie.fr (Righetti), pour le ton "maison d'art verrier" plutôt que "artisan du bâtiment" ; et plus largement des sites vitrines "digital agency" ou "architecture" primés (type Awwwards) pour le niveau de finition attendu, indépendamment du secteur

### Animations & fluidité

- **Scroll fluide** avec effet de parallaxe léger sur le hero et certaines images de section
- **Apparition progressive des éléments au scroll** (fade + léger déplacement vertical) — pas d'un coup, en cascade sur les grilles (cartes services, galerie)
- **Micro-interactions** sur les éléments interactifs : hover sur les cartes de services (légère élévation + zoom image), hover sur les boutons (transition de couleur/fond fluide), transitions de filtre dans la galerie (fade/morph entre les images)
- **Curseur personnalisé** possible sur desktop pour renforcer l'aspect premium (optionnel — à tester, ne pas casser l'accessibilité mobile)
- Toutes les animations doivent rester **subtiles et rapides** (200-500ms) — l'objectif est la sensation de fluidité, jamais la lenteur ou la surcharge visuelle
- Respecter `prefers-reduced-motion` pour l'accessibilité

### Élément(s) 3D

Prévoir **au moins une scène 3D interactive** qui exploite le matériau du métier (verre/reflet/lumière) — pistes possibles, à choisir/adapter selon faisabilité :
- Un **objet 3D de verre/miroir stylisé** dans le hero qui réagit légèrement au mouvement de la souris (parallaxe 3D, rotation douce)
- Un **panneau de verre 3D avec effet de réfraction/reflet réaliste** en arrière-plan d'une section clé
- Une transition 3D subtile entre deux sections (effet de "traversée du verre")

Cet élément doit être un vrai atout visuel, pas un gadget technique qui ralentit le site — privilégier **une seule scène 3D bien exécutée** plutôt que plusieurs éléments 3D moyens. Prévoir un fallback statique (image) sur mobile/basse performance si nécessaire pour ne pas pénaliser les temps de chargement.

## 6. STACK TECHNIQUE

- **Next.js** (App Router) + TypeScript
- **Tailwind CSS**
- **Framer Motion** pour les animations d'interface (apparitions au scroll, micro-interactions, transitions)
- **React Three Fiber** (+ Three.js / drei) pour l'élément 3D — bibliothèque recommandée car bien intégrée à l'écosystème React/Next.js, permet un fallback statique propre si besoin de performance
- Contenu (services, textes, items de galerie) géré via **fichiers de données structurés** (ex. `/data/services.ts`, `/data/gallery.ts`) — pas de CMS, les mises à jour se feront directement dans le code
- `next/image` pour l'optimisation des photos
- Formulaire : intégration email (Resend ou équivalent) + génération lien `wa.me` avec message pré-rempli
- Déploiement : **Contabo VPS** (self-hosted, pas de Vercel)

### Déploiement Contabo VPS

- **Build Next.js** en mode `standalone` (`output: 'standalone'` dans `next.config`) — image Docker légère, prête pour la prod
- **Docker + Docker Compose** sur le VPS : conteneur Node pour l'app Next.js
- **Nginx** en reverse proxy devant l'app (port 80/443 → conteneur sur port interne, ex. 3000)
- **SSL** : Let's Encrypt via Certbot (HTTPS obligatoire)
- **Domaine** : DNS A record pointant vers l'IP du VPS Contabo
- **Variables d'environnement** sur le serveur (`.env` ou secrets Docker) : clé API email (Resend), numéros WhatsApp/téléphone, etc. — jamais commitées dans le repo
- **Mises à jour** : pull Git sur le VPS → rebuild Docker → redémarrage du conteneur (script `deploy.sh` ou pipeline CI simple)
- **Pas de base de données** — le site reste entièrement statique côté contenu (fichiers `/data/`)
- **Optimisation images** : installer `sharp` côté serveur pour que `next/image` fonctionne en self-hosted (contrairement à Vercel qui le gère nativement)

### Résilience & haute disponibilité (VPS)

Objectif : si l'app plante, redémarre ou est en cours de déploiement, **le visiteur ne doit pas voir une erreur brute** (502 Bad Gateway, page blanche, etc.). Nginx reste toujours en ligne et gère les cas de panne.

#### 1. Redémarrage automatique

- **Docker Compose** : `restart: unless-stopped` sur le conteneur Next.js — redémarrage auto après crash ou reboot du VPS
- **Docker au boot** : activer le service Docker au démarrage du serveur (`systemctl enable docker`) pour que les conteneurs remontent seuls après un redémarrage Contabo
- **Healthcheck Docker** : ping régulier de `/api/health` — si l'app ne répond plus, Docker relance le conteneur automatiquement

#### 2. Endpoint de santé (`/api/health`)

- Route API légère qui renvoie `200 OK` quand l'app est prête
- Utilisée par Docker healthcheck et par le monitoring externe
- Exemple de réponse : `{ "status": "ok", "timestamp": "..." }`

#### 3. Nginx — toujours en ligne, jamais d'erreur nue

Nginx tourne **en dehors** du conteneur app (service système ou conteneur séparé). Même si Next.js est down, Nginx répond.

- **Page de maintenance statique** (`/var/www/maintenance/index.html`) — design sobre aux couleurs du site (logo, message "Site momentanément indisponible", lien WhatsApp direct). Servie par Nginx quand l'upstream Next.js est injoignable
- **Pages d'erreur personnalisées** pour 502, 503, 504 — même fallback maintenance ou page d'erreur brandée (pas la page nginx par défaut)
- **Config Nginx** :
  - `proxy_connect_timeout`, `proxy_read_timeout` — timeouts raisonnables (ex. 30s)
  - `proxy_next_upstream error timeout` — retry sur un second upstream si configuré
  - `error_page 502 503 504 /maintenance.html` — redirection vers la page statique
- **Upstream avec vérif** : `max_fails` + `fail_timeout` sur le bloc `upstream` pour détecter rapidement un backend mort

#### 4. Déploiement sans coupure visible

Le script `deploy.sh` doit minimiser la fenêtre où l'app est indisponible :

1. Build de la **nouvelle image** Docker (l'ancienne continue de tourner)
2. Bascule : `docker compose up -d --no-deps --build app` — remplace le conteneur une fois le build prêt
3. Attendre que `/api/health` réponde `200` avant de considérer le deploy terminé
4. Si le healthcheck échoue après deploy → **rollback automatique** vers l'image précédente (tag `:previous` conservé à chaque deploy)
5. Pendant les ~10–30s de bascule, Nginx sert la **page de maintenance** au lieu d'une 502

#### 5. Monitoring & alertes

- **UptimeRobot** (gratuit) ou **Uptime Kuma** (self-hosted sur le même VPS) — ping `/api/health` toutes les 5 min
- Alerte email/Telegram si le site ne répond plus → intervention avant que le client ne le remarque
- Logs centralisés : `docker compose logs -f app` + rotation des logs (`logrotate` ou limite Docker `max-size`)

#### 6. Sauvegardes minimales

- Pas de DB, mais sauvegarder : config Nginx, `.env`, certificats SSL (`/etc/letsencrypt`), script deploy
- Snapshot Contabo mensuel (option VPS) — filet de sécurité en cas de corruption serveur

#### Fichiers infra à prévoir

```
/
├── Dockerfile
├── docker-compose.yml       # restart policy + healthcheck
├── deploy.sh                  # build → switch → healthcheck → rollback
├── nginx/
│   ├── site.conf              # reverse proxy + error pages + maintenance
│   └── maintenance/
│       └── index.html         # page statique fallback
└── app/api/health/route.ts    # endpoint de santé
```

**Point d'attention performance :** les librairies 3D/animation ne doivent jamais faire chuter le score Lighthouse en dessous d'un niveau correct (viser 85+ sur mobile) — un site "premium" qui rame perd tout son effet.

## 7. IMAGES NÉCESSAIRES

Le client a un catalogue papier scanné (photos pliées, qualité moyenne, ancien). Pour un rendu premium il faut :
- Idéalement : nouvelles photos professionnelles des réalisations existantes (au moins les pièces phares : coupole vitrail, panneaux miroir bronze, escaliers verre)
- À défaut : demander les fichiers photo originaux (non scannés) au client — ils existent probablement sur un téléphone ou dans un dossier
- En attendant : possibilité de démarrer le développement avec des placeholders/images génériques de haute qualité (verre, miroir, architecture) pour valider la structure et le design, puis remplacer par les vraies photos avant mise en ligne
